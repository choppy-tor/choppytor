# Tor Client Instrumentation for Experimental Circuit Tagging

This document describes the client-side changes made to Tor for the experimental artifact. The purpose of these changes is to explicitly tag circuits carrying different application traffic types, such as VoIP and web traffic, so that relay-side instrumentation can later associate scheduler events with the correct circuit.

The implementation was done on Tor `0.4.8.21`.

---

## 1. Objective of the Client-Side Changes

The Tor client was modified to send a one-time experimental tag on each circuit used in our measurements.

The tag allows the instrumented middle relay to identify whether a circuit corresponds to:

- VoIP traffic
- Web traffic
- Unknown or unclassified traffic

After the middle relay receives this tag, subsequent scheduler events can be tracked using the circuit identifier logged at the relay. The tag is not sent with every cell. It is sent only once per circuit.

---

## 2. High-Level Design

The client-side modification adds a custom relay command called:

```c
RELAY_COMMAND_EXPTAG
```

This command carries a small experimental payload containing:

```c
#define EXPTAG_MAGIC0 'X'
#define EXPTAG_MAGIC1 'P'
#define EXPTAG_MAGIC2 'T'
#define EXPTAG_MAGIC3 'G'
```

The magic string `XPTG` is used to distinguish our experimental tag from normal relay payloads.

A typical payload contains:

| Field | Purpose |
|---|---|
| Magic bytes | Identifies the payload as an experimental tag |
| Traffic type | Indicates whether the circuit is VoIP, web, or unknown |
| Tag identifier | Unique identifier used to correlate logs |

The command is sent from the Tor client toward the selected relay on the circuit. In our experiments, the middle relay is instrumented to receive and log this command.

The main client-side changes are made in Tor's relay/circuit handling path.


---

## 4. Adding a New Relay Command

The custom relay command is defined in Tor's relay command definitions.

Example:

```c
#define RELAY_COMMAND_EXPTAG 0x7F
```

The value should be chosen from an unused relay command number for the private experimental build. This command is not part of the public Tor protocol and should not be used on the public Tor network.

The command is used only in the private/instrumented testbed.

---

## 5. Adding Client-Side Circuit Metadata

To avoid sending the tag repeatedly, we added metadata to the origin circuit structure.

Conceptually, each client-side circuit maintains:

```c
int exptag_sent;
int exptag_type;
uint64_t exptag_id;
```

The fields have the following meaning:

| Field | Meaning |
|---|---|
| `exptag_sent` | Whether the experimental tag has already been sent on this circuit |
| `exptag_type` | Traffic type associated with the circuit |
| `exptag_id` | Random or sequential identifier for correlating logs |

This ensures that the tag is sent only once for each circuit.

---

## 6. Identifying VoIP and Web Circuits at the Client

The Tor client classifies circuits based on the application traffic that created the stream.

In our artifact, the classification is based on the client-side setup used for the experiment. For example:

- VoIP traffic is generated by the caller/callee scripts through a known SOCKS/OpenVPN path.
- Web traffic is generated by automated browsing or parallel web-flow scripts.
- The client assigns a traffic type before or when the stream is attached to a circuit.

The classification is then stored in the corresponding origin circuit.

Conceptual traffic-type values:

```c
#define EXPTAG_TYPE_NONE 0
#define EXPTAG_TYPE_VOIP 1
#define EXPTAG_TYPE_WEB  2
```

---

## 7. Where the Tag Is Sent

The tag is sent after the client has identified the circuit and before or during the early use of the circuit.

The relevant client-side call path is the stream-to-circuit attachment path. In Tor, this usually involves functions such as:

```c
connection_ap_handshake_attach_chosen_circuit()
connection_ap_handshake_attach_circuit()
connection_edge_package_raw_inbuf()
```

The exact hook point depends on where the traffic type is assigned. In our artifact, the tag is sent only after the client knows which circuit is carrying the corresponding traffic.

The helper function added for sending the tag can be documented as:

```c
static void
exptag_send_circuit_tag(origin_circuit_t *circ, int tag_type)
{
    if (!circ)
        return;

    if (circ->exptag_sent)
        return;

    uint8_t payload[EXPTAG_PAYLOAD_LEN];

    payload[0] = EXPTAG_MAGIC0;
    payload[1] = EXPTAG_MAGIC1;
    payload[2] = EXPTAG_MAGIC2;
    payload[3] = EXPTAG_MAGIC3;

    payload[4] = (uint8_t) tag_type;

    /* Remaining bytes encode the tag identifier. */

    relay_send_command_from_edge(
        0,
        TO_CIRCUIT(circ),
        RELAY_COMMAND_EXPTAG,
        (const char *)payload,
        EXPTAG_PAYLOAD_LEN,
        NULL
    );

    circ->exptag_sent = 1;
}
```

The important point is that the experimental relay command is injected into the circuit using Tor's existing relay-cell sending path.

---

## 8. Why the Tag Is Sent Only Once

The tag is not intended to carry traffic information for every packet or cell. It is only used to label the circuit.

Once the middle relay receives the tag, it logs the local circuit pointer or circuit identifier along with the tag type. After that, later scheduler events can be associated with the same circuit using the circuit identifier.

This avoids unnecessary overhead and prevents the tag from affecting the scheduling behavior being measured.

---

## 9. Relay-Side Interpretation

Although this document focuses on the client, the client-side tag is meaningful only because the relay has corresponding instrumentation.

At the relay, the custom command is handled in the relay-cell processing path, typically near:

```c
handle_relay_cell_command()
```

When the relay receives `RELAY_COMMAND_EXPTAG`, it parses the payload, checks the magic bytes, extracts the traffic type, and marks the local circuit as VoIP or web.

Example relay-side log line:

```text
EXPTAG middle recv circ=0x56c016a92890 type=web id=13498108163233928593
```

After this initial log, later events such as active, inactive, pick, and xmit can be tracked using the same circuit identifier.

---

## 10. Logging Behavior

The client-side tag itself is sent once per circuit.

The relay-side logs then track scheduler events. These logs do not need to contain the traffic type in every line. Instead, the analysis script first identifies the circuit using the initial `EXPTAG` log and then maps later events using the circuit identifier.

Example scheduler log format:

```text
ewma_notify_circ_active(): EWMAA t_us=... ev=active circ=... side=p a=... qsize=...
```

The analysis pipeline reconstructs the traffic type by joining these later scheduler events with the initial circuit-tag log.

---

## 11. Directionality

The tag labels the circuit at the relay, but the circuit has two directions. In Tor relay logs, the direction is typically represented using fields such as:

```text
side=p
side=n
```

The tag itself is not repeatedly sent in both directions. Instead, once a circuit is identified, events on both sides of the circuit can be analyzed using the same local circuit object and side field.

This is sufficient for our analysis because we are interested in when cells become active, are selected by the scheduler, and are transmitted from relay queues.

---

## 12. Artifact-Relevant Summary

The Tor client changes can be summarized as follows:

1. Add a private relay command `RELAY_COMMAND_EXPTAG`.
2. Add a small payload format beginning with the magic string `XPTG`.
3. Add per-origin-circuit metadata to record whether a tag has already been sent.
4. Classify circuits as VoIP or web based on the experiment setup.
5. Send the tag once per circuit using Tor's relay-cell sending path.
6. Let the instrumented middle relay log the tag and associate future scheduler events with the circuit.

These changes do not modify Tor's anonymity protocol for normal use and are intended only for controlled experimental measurements.

The Tor client was modified to send a one-time experimental relay command on circuits used by the measurement applications. The command carries a small payload beginning with the magic string `XPTG`, followed by a traffic-type field and a tag identifier. This allows the instrumented middle relay to distinguish VoIP and web circuits. The tag is sent only once per circuit; subsequent relay-side scheduler events are associated with the traffic type by matching the logged circuit identifier. This design keeps the instrumentation lightweight and avoids injecting additional per-cell measurement traffic.
